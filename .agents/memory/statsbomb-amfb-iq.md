# StatsBomb AmFB IQ GraphQL API

Separate source from Telemetry/Hudl Wire. Used for rich RAW player stats
(offense/defense/special-teams + physical/speed), NOT derived grades.

## Endpoint & auth
- Endpoint: `https://amf-iq-api.statsbomb.com/api` (GraphQL POST).
  - The docs/AI-search say `http://amf-iq-api-statsbomb.com/api` but that host does NOT resolve. Use the subdomain form above.
- Auth header: `Authorization: apikey <STATSBOMB_API_KEY>`. Key generated by the user on `https://iq-amf.statsbomb.com/data-access`. Stored as secret `STATSBOMB_API_KEY`.
- **The code_execution sandbox redacts secret values to the boolean `true`** — you cannot read/test a secret value there. Test from bash/`curl` using `$STATSBOMB_API_KEY` (never print it) or from server code via `process.env`.
- Error shape is REST-style `{title, description}` even for GraphQL errors; GraphQL field errors come back as `{errors:[...]}`.

## Key queries
- `competitions { id name level }` → NCAA is `id 1446`.
- `seasons(competitionId:1446) { id name }` → name is `"YYYY/YYYY+1"`. Fall season year Y maps to name `"Y/Y+1"`. e.g. 2019→id 42, 2020→90, 2021→224, 2022→1000, 2023→1034, 2024→1052, 2025→1085, 2026→1118.
- `playerStats(aggregateMode:TOTAL, groupBy:[PLAYER], playFilters:$pf){ totalCount items { ... } }` with `$pf:PlayFilter!` = `{competitionId:1446, seasonId:<id>}`. Returns ~11k items (one per NCAA player) for a season.
  - `limit`/`offset` args exist in the schema but the resolver REJECTS `limit` ("unexpected keyword argument 'limit'"). Do not pass them; fetch the whole season.
  - `playerStatFilters:PlayerStatFilter` supports `teamIds`, `playerName`, `minPlays`, `specificPositions`, etc. to narrow.
- `playerStatDefinitions { name prettyName categories subcategory rate units abbrev lowerIsBetter }` → 510 stat metadata rows. `name` is the GraphQL field; `prettyName` is the label; `units` ∈ COUNT/YARDS/PERCENTAGE/EPA/…; `categories` are position groups; `subcategory` groups within.

## The 3-group constraint (critical)
A single `playerStats` query **cannot mix special-teams, physical, and "normal" (offense/defense) stat fields** — server rejects with "playerStats query cannot mix special teams, physical and normal stats". Run THREE separate queries per season and merge by player:
- **Special Teams**: defs whose `categories` includes `"Special Teams"`.
- **Physical**: defs whose `categories` includes `"Physical"` (speed/accel/get-off).
- **Normal**: everything else (incl. context fields like `plays`, `games`, `odSnaps` which have empty categories).
Identity/context fields (`playerName`, `teamName`, `teamShortName`, `rosterPosition`, `playPosition`, `jerseyNumber`) are NOT stats and can be selected in every group query; use them to merge the 3 results.

## Fetch per-team, not whole-season (critical)
A full-season unfiltered `playerStats` query (just `{competitionId, seasonId}`) errors server-side at scale. In practice you MUST iterate teams and pass `playerStatFilters.teamIds` per request, looping over all ~300 NCAA teams for the season, then merge. This makes the sync slow (~5-7 min total) but is the only reliable path. A successful 2025 sync writes ~471k stat lines for matched FBS players.

## Player matching (the hard part)
`PlayerStat` items have NO shared external player id with our DB — match by name+team. Our `players` table stores `player_name` and `team` (plain school string, e.g. "LSU"). Match on normalized(playerName)+normalized(team); StatsBomb `teamName`/`teamShortName` may need an alias map to our `school` values. Treat as best-effort and log match-rate during sync.
